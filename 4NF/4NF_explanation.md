4NF Transformation, Group 11

A table is in 4NF once it is already in BCNF and it does not store two separate many to many facts about the same thing in one place. The easy way to think about it is this. If an entity has two lists that have nothing to do with each other, those two lists need to live in two different tables. If they end up sharing a table, every value in one list gets repeated next to every value in the other, and you end up with rows that never actually happened in real life.

I went through every table that came out of BCNF and checked for that exact problem. Here is what I found and how I checked it.

The first thing I noticed was bigger than the textbook worker example. The raw data has one row per project and worker combination, and on every one of those rows it also repeats that project's supplier, material, and equipment details. A project's list of workers has nothing to do with its list of suppliers, or its list of materials, or its list of equipment. Those are four separate facts about the same project, all stuffed into the same row. That is the real four NF violation in this dataset, and it shows up the moment you look at how many times the same supplier name or material name gets copied across rows that only differ by which worker is listed.

The second one is the example everyone expects, which is the worker's skills and certifications. A worker's skills have nothing to do with that same worker's certifications, so combining them in one row also creates the same kind of false combination problem. When the 1NF stage exploded both of those columns on the same row, a worker with two skills and two certifications turned into four rows instead of the four real facts that were actually being recorded.

One thing that is not a violation here is supplier phone numbers. A supplier can have more than one phone number, but that list was never mixed with a second independent list for the same supplier, so on its own it only ever needed fixing back at 1NF, not here.

By the time the BCNF tables were finished, both of those problems were already gone. Project workers, project suppliers, project materials, and project equipment are four separate tables, each with its own key. Worker skills and worker certifications are also two separate tables. So my job at this stage was mostly to confirm that split actually held up across every table, not to redo it.

I checked all fourteen tables one by one. Projects, clients, supervisors, workers, suppliers, equipment, and materials hold single valued facts only, so none of them have a many to many problem to begin with. Project workers holds exactly one independent list, the workers on a project, and nothing else is mixed in. The same goes for project suppliers, project materials, and project equipment, each one holds exactly one list tied to its key, plus a cost column where that applies, and that cost depends on the full key rather than being a second independent list. Worker skills and worker certifications each hold exactly one list as well, tied to the worker. None of the fourteen tables hold two unrelated lists at once, so none of them needed to be split any further.

That means the 4NF folder ends up holding the same fourteen tables that came out of BCNF, unchanged. That is not me skipping the step, that is the actual result of checking it properly. Pulling project workers, project suppliers, project materials, and project equipment back together by joining on ProjectID gives you exactly the original set of project associations, nothing extra and nothing missing, which is the actual test for whether a split like this was done correctly.

Files in this folder: projects, clients, supervisors, workers, suppliers, equipment, materials, project workers, project suppliers, project materials, project equipment, worker skills, worker certifications, supplier phones. All fourteen are carried over from the BCNF folder with no changes, because the audit above did not find anything left to fix.
